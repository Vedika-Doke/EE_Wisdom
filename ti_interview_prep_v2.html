What is CAN? (Controller Area Network)

  • Invented by: Bosch (1986), now ISO 11898 standard
  • Why it exists: Automotive ECUs need reliable, real-time communication in electrically noisy environments (ignition, motors, alternators)
  • Physical layer: Differential signaling on 2 wires (CAN-H, CAN-L)
  • Data rate: Up to 1 Mbps (you used 500 kbps)
  • Topology: Multi-drop bus — all nodes connected to same 2 wires
  • Termination: 120Ω resistors at each end of the bus (you have this in your PCB!)

Differential Signaling — Why CAN is noise-immune

Dominant (logic 0): CAN-H = 3.5V CAN-L = 1.5V Diff = 2.0V Recessive (logic 1): CAN-H = 2.5V CAN-L = 2.5V Diff = 0.0V CAN-H ──────┐ ┌────── 3.5V │ │ 2.5V --------+------------+-------- (idle/recessive) 1.5V ────────┘ └────── CAN-L

Electromagnetic noise couples equally into both wires (common-mode). The receiver reads the difference (CAN-H minus CAN-L), so the noise cancels out. This is why CAN works reliably in a race car with high-current motors nearby.

CAN Frame Structure

| SOF | Identifier (11-bit) | RTR | IDE | r0 | DLC (4-bit) | Data (0-8 bytes) | CRC (15-bit) | ACK | EOF | SOF = Start of Frame (dominant bit) Identifier = Message priority (lower = higher priority) RTR = Remote Transmission Request (0 = data frame, 1 = remote request) IDE = Identifier Extension (0 = standard 11-bit, 1 = extended 29-bit) DLC = Data Length Code (how many data bytes, 0-8) CRC = Cyclic Redundancy Check (error detection) ACK = Acknowledgment slot (receiver drives dominant to confirm receipt) EOF = End of Frame (7 recessive bits)

Arbitration — How priority works

If two nodes try to transmit simultaneously, they both start sending their identifier bit by bit. Dominant (0) overwrites recessive (1). The node with the lower ID wins because it keeps seeing its own bits, while the losing node detects a dominant bit where it sent recessive and backs off.

Example: Node A sends ID 0x201 (0010 0000 001) Node B sends ID 0x202 (0010 0000 010) At bit 1 (from right): A sends 1 (recessive), B sends 0 (dominant) Bus shows 0 → A sees mismatch, backs off. B wins. Lower ID = higher priority (your right motor 0x201 has priority over left 0x202)

Error Detection in CAN (5 mechanisms)

  1. CRC check — 15-bit polynomial over the frame
  2. Bit stuffing — after 5 consecutive identical bits, a stuff bit of opposite polarity is inserted. If receiver sees 6 identical bits, it's a stuff error
  3. Frame check — fixed-form fields (EOF, ACK delimiter) must have expected values
  4. ACK check — transmitter checks that at least one node acknowledged
  5. Bit monitoring — transmitter reads back the bus; if it sent recessive but reads dominant (outside arbitration), it's an error

Q: Walk me through what happens when the driver presses the accelerator.
Two redundant APPS sensors read the pedal position as analog voltages. The Teensy's ADC converts these to 10-bit values (0-1023). Firmware normalizes both to 0.0-1.0 range using calibrated min/max values. It then checks for faults: hardware failure (values at ADC rails), sensor disagreement (>10% difference for >100ms), and brake/throttle plausibility (both pressed for >1s). If all checks pass, it averages the two values, multiplies by a torque scaler (0x90), and sends the torque command as a CAN frame to both motor controllers at IDs 0x201 and 0x202.
Q: Why CAN bus and not I2C or SPI?
CAN is designed for automotive environments — its differential signaling rejects electromagnetic noise from motors and power electronics, it supports multi-node communication on a simple 2-wire bus (I2C also uses 2 wires but can't handle noise or long distances), it has built-in error detection (CRC + bit stuffing + ACK), and priority-based arbitration ensures critical messages get through first. SPI would need a separate chip-select for each node, and I2C can't handle the noise or cable lengths in a car. CAN is the industry standard for vehicle ECUs.
Q: What is bit stuffing in CAN?
After 5 consecutive bits of the same value, the transmitter inserts one bit of the opposite value. The receiver knows to remove these stuff bits. Purpose: it maintains synchronization (long runs of identical bits could cause clock drift), and it provides error detection (6 consecutive identical bits = stuff error).
Q: What happens if a CAN message is corrupted?
The receiving node's CRC check fails, so it sends an Error Frame (6 dominant bits, violating bit stuffing). All nodes see this and discard the message. The transmitting node automatically retransmits. CAN also has error counters — if a node causes too many errors, it progressively moves from Error Active to Error Passive to Bus Off, isolating the faulty node.
Q: Why do you need 120Ω termination resistors?
The CAN bus is a transmission line. Without proper termination, signals reflect off the open ends of the wire and cause interference. The 120Ω resistors at each end of the bus match the characteristic impedance of the twisted pair cable (~120Ω), absorbing the signal and preventing reflections. Two 120Ω in parallel = 60Ω total bus resistance.

The Problem

Two identical temperature sensors (same manufacturer, same part) needed to be on the same I2C bus. But they both have the same hardcoded I2C address (0x48). When the master sends a read request to 0x48, BOTH sensors respond simultaneously, corrupting the data on SDA.

Why not just change the address? Some I2C devices have address pins (A0, A1) you can tie HIGH/LOW to set different addresses. These particular sensors had no such option — the address is burned into the chip.

The Solution: TCA9543A I2C Multiplexer

┌─── Channel 0 ─── [Temp Sensor 1] (addr 0x48) Teensy ──I2C──── TCA9543A ──┤ └─── Channel 1 ─── [Temp Sensor 2] (addr 0x48) TCA9543A's own address: 0x70 (configurable via A0, A1 pins)

The TCA9543A is a 2-channel I2C switch (made by TI!). It sits between the master and the sensors. You write a control byte to the mux (at address 0x70) to select which downstream channel is active. Only one channel is connected at a time, so the address conflict disappears.

// Firmware: Channel selection logic #include <Wire.h> #define MUX_ADDR 0x70 void selectChannel(uint8_t channel) { Wire.beginTransmission(MUX_ADDR); Wire.write(1 << channel); // bit 0 = ch0, bit 1 = ch1 Wire.endTransmission(); } float readTemp(uint8_t channel) { selectChannel(channel); // switch mux to this sensor Wire.requestFrom(0x48, 2); // read 2 bytes from temp sensor int16_t raw = (Wire.read() << 8) | Wire.read(); return raw * 0.0625; // convert to Celsius } // In main loop: float temp1 = readTemp(0); // sensor on channel 0 float temp2 = readTemp(1); // sensor on channel 1

Level Shifting with TXS0102

The Teensy runs at 3.3V but some sensors need 5V I2C. The TXS0102 is a bidirectional voltage-level translator — it translates 3.3V signals to 5V and vice versa, on both SDA and SCL lines simultaneously.

Why needed: If you connect a 3.3V device directly to a 5V I2C bus, the 5V signal could damage the 3.3V device, and the 3.3V HIGH might not be recognized as HIGH by the 5V device (5V logic threshold is ~3.5V for some chips).

Why Instrumentation Amplifier?

Strain gauges and load cells output millivolt-level differential signals (typically 1-10 mV at full scale). The Teensy's ADC range is 0-3.3V. You need ~500x gain, with high precision and noise rejection.

  • INA333 is a precision instrumentation amp (made by TI!)
  • Gain equation: G = 1 + (100kΩ / R_G) — set by a single external resistor R_G
  • CMRR: >100 dB — rejects common-mode noise (vibration, EMI)
  • Why not a regular op-amp? An instrumentation amp has matched input impedances on both inputs, very high CMRR, and single-resistor gain setting. A regular differential amp would need precisely matched resistors for good CMRR.

The trimpots (Bourns 3296W) allow fine calibration of gain and offset in the field.

Q: How does I2C work?
I2C uses 2 wires: SDA (data) and SCL (clock), both with pull-up resistors to VCC. Communication: Master sends a START condition (SDA falls while SCL is HIGH), then sends the 7-bit slave address + R/W bit. The addressed slave pulls SDA LOW for ACK. Then data transfers in 8-bit chunks, each followed by an ACK. Master sends STOP (SDA rises while SCL is HIGH). Multiple devices share the same bus, each with a unique address — which is exactly why our identical-address sensors were a problem.
Q: What is an instrumentation amplifier and when would you use one?
An instrumentation amp is a precision differential amplifier with very high input impedance on both inputs, extremely high CMRR (>100dB), and gain set by a single external resistor. You use it when you need to amplify small differential signals in the presence of large common-mode noise — exactly what we had with strain gauges in a vibrating race car. The INA333 we used has gain G = 1 + 100k/R_G.

  • Voltage method: Measure open-circuit voltage, look up SOC from the V-SOC curve. Simple but inaccurate under load (voltage sags due to internal resistance). Must correct for current and temperature.
  • Coulomb counting: Integrate current over time: SOC(t) = SOC(0) - (1/Q)∫I dt. Accurate short-term but drifts over time due to accumulated sensor error.
  • Kalman Filter: Combines voltage method + coulomb counting. Uses a state-space model of the battery. Predicts SOC from coulomb counting, then corrects using voltage measurement. Best accuracy but most computationally expensive.

Why Balancing is Needed

In a series string, all cells see the same current but have slightly different capacities and self-discharge rates. Over charge/discharge cycles, their SOCs diverge. Without balancing, the weakest cell hits its voltage limit first, limiting the entire pack.

Passive Balancing (your team's approach)

Drain higher-SOC cells through a bleed resistor until they match the lowest cell. Done during charging only. Simple and cheap, but wastes energy as heat.

Active Balancing

Transfer charge from higher-SOC cells to lower-SOC cells using inductors or capacitors. More efficient but much more complex PCB and control logic.

Torque Vectoring System

IITB Racing — Trainee Project — MATLAB/Simulink → STM32

Three Degrees of Freedom

  1. Longitudinal (x): Forward/backward acceleration — governed by total torque and drag
  2. Lateral (y): Side-to-side forces during cornering — governed by tire lateral forces
  3. Yaw (ψ): Rotation about vertical axis — this is what torque vectoring controls
Yaw dynamics: I_z * dψ/dt = (F_right - F_left) * track_width / 2 where: I_z = moment of inertia about vertical axis F_right, F_left = longitudinal forces at each wheel track_width = distance between left and right wheels Desired yaw rate: ψ_desired = V * tan(δ) / (L * (1 + K_us * V²)) where δ = steering angle, L = wheelbase, K_us = understeer gradient

Control Loop

┌──────────────────────────────────┐ │ │ ψ_desired ──(-)──→ PID ──→ ΔTorque ──→ Vehicle ──→ ψ_actual ↑ error │ │ │ └──────────── Yaw Rate Sensor ←────────────┘ (via CAN bus)
error = ψ_desired - ψ_actual ΔTorque = Kp * error + Ki * ∫error dt + Kd * d(error)/dt Left_torque = Base_torque - ΔTorque / 2 Right_torque = Base_torque + ΔTorque / 2

What each term does:

  • P (Proportional): Immediate correction proportional to error. Higher Kp = faster response but can overshoot. If the car understeers (yaw rate too low), P immediately adds more torque to the outer wheel.
  • I (Integral): Accumulates error over time, eliminates steady-state offset. Without I, the car might have a persistent small yaw error during constant-radius turns.
  • D (Derivative): Responds to rate of change of error. Dampens oscillations. Prevents the correction from overshooting when the car transitions between turns. Critical during chicanes and slaloms.

Understeer vs Oversteer Correction

  • Understeer: Actual yaw rate < desired (car turns less than steering commands). Error is positive. Controller adds more torque to the outer wheel, creating a yaw moment that turns the car more.
  • Oversteer: Actual yaw rate > desired (car rotates too much, tail slides out). Error is negative. Controller adds more torque to the inner wheel, counteracting the excessive rotation.
  1. Model in Simulink: Build the vehicle dynamics + PID controller as block diagrams. Simulate with various steering inputs and speeds. Tune PID gains until performance is satisfactory.
  2. Embedded Coder: MATLAB's Embedded Coder generates optimized, production-quality C code from the Simulink model. It handles fixed-point conversion, loop unrolling, and memory optimization for the target MCU.
  3. Compile & Flash: Generated C code is compiled with STM32 toolchain (arm-none-eabi-gcc) and flashed onto the STM32 microcontroller.
  4. Sensor Integration: Yaw rate sensor sends data over CAN bus. STM32 reads CAN frames, feeds yaw rate into the PID controller, computes torque split, sends commands to motor controllers.

Key advantage: Same mathematical model validated in simulation runs on real hardware. Reduces sim-to-real gap dramatically compared to hand-coding the controller.

How the Channel Forms

  1. V_GS = 0: Two back-to-back pn junctions (n+-p-n+). No current can flow. Device is OFF.
  2. 0 < V_GS < V_th: Positive gate voltage repels holes from the surface, creating a depletion region under the gate. Still no continuous n-channel. Only subthreshold leakage current flows (exponential with V_GS).
  3. V_GS = V_th: Surface electron concentration equals substrate hole concentration — this is onset of inversion. A thin n-type channel connects source to drain.
  4. V_GS > V_th: Strong inversion. Dense n-channel exists. Current can flow from drain to source (electrons flow source to drain).

1. Cutoff (V_GS < V_th)

I_D = 0 (ideally) Subthreshold: I_D ∝ exp(V_GS / nV_T) where n ≈ 1-2, V_T ≈ 26mV

No channel. Transistor is OFF. Used as a switch in the open state.

2. Linear / Triode (V_GS > V_th AND V_DS < V_GS - V_th)

I_D = μ_n C_ox (W/L) [(V_GS - V_th)V_DS - V_DS²/2] For small V_DS: I_D ≈ μ_n C_ox (W/L)(V_GS - V_th)V_DS ← linear like a resistor R_on = 1 / [μ_n C_ox (W/L)(V_GS - V_th)]

Channel exists from source to drain, uniform thickness. MOSFET behaves like a voltage-controlled resistor. Current increases linearly with V_DS (for small V_DS). Used as a switch in the closed state or as a variable resistor.

3. Saturation (V_GS > V_th AND V_DS ≥ V_GS - V_th)

I_D = (μ_n C_ox / 2)(W/L)(V_GS - V_th)² × (1 + λV_DS) V_ov = V_GS - V_th (overdrive voltage) I_D ∝ V_ov² ← square-law device

Channel is pinched off at the drain end (where V_GD = V_th). Current is controlled by V_GS only (ideally constant with V_DS). This is the amplification region — used in analog circuits. The λV_DS term models channel length modulation (finite output resistance).

Memory aid: Cutoff = switch OFF, Linear = switch ON (resistor), Saturation = amplifier. The boundary between linear and saturation is V_DS = V_GS - V_th (= V_ov).

What determines V_th?

V_th = V_FB + 2φ_F + Q_dep/C_ox V_FB = flat-band voltage (depends on gate material work function & oxide charges) φ_F = Fermi potential (depends on substrate doping: φ_F = V_T ln(N_A/n_i)) Q_dep = depletion charge (depends on doping and body bias) C_ox = ε_ox / t_ox (oxide capacitance per unit area)

Factors: gate oxide thickness (↑t_ox → ↑V_th), substrate doping (↑N_A → ↑V_th), body effect (V_SB > 0 → ↑V_th), gate material, oxide charges.

Channel Length Modulation

In saturation, ↑V_DS pushes pinch-off point away from drain, shortening effective channel. This makes I_D increase slightly with V_DS instead of being perfectly flat.

I_D = (K/2)(V_GS - V_th)² (1 + λV_DS) λ ∝ 1/L (shorter channel → larger λ → less ideal) Output resistance: r_o = 1/(λI_D)

NAND: NMOS series + PMOS parallel   |   NOR: NMOS parallel + PMOS series
Pull-up is always the dual of pull-down (series↔parallel, NMOS↔PMOS).
N-input gate = 2N transistors.

Why NAND > NOR in CMOS

NAND has PMOS in parallel (fast) + NMOS in series (OK since electrons are fast). NOR has PMOS in series (very slow — holes are 2-3× slower + series R compounds). NAND is ~1.5-2× faster → universal gate of choice.

Q: Design a 2:1 MUX. How many transistors?
6T. Two transmission gates (4T) + inverter for S̄ (2T). TG1 passes A when S=0, TG2 passes B when S=1. Full rail-to-rail, no threshold drop. This is the standard cell approach.
Q: Why zero static power in CMOS?
Pull-up and pull-down are complementary — one is always OFF. No DC path from VDD to GND in steady state. Power only during switching (dynamic) and brief short-circuit during transitions.
Q: Implement Y = (AB + C)' in CMOS.
PDN (NMOS): (A series B) parallel C. PUN (PMOS): dual → (A parallel B) series C. 6 transistors total.
Q: Transmission gate vs single pass transistor?
Single NMOS loses Vtn passing a 1. Single PMOS loses |Vtp| passing a 0. TG (both in parallel) gives full rail-to-rail — NMOS handles lows, PMOS handles highs. Trade-off: needs complementary control (S and S̄).
Q: Why PMOS wider than NMOS?
μp ≈ (1/2 to 1/3) × μn. Equal rise/fall times require Wp = 2-3 × Wn to match on-resistance.
Q: Transistor count for a D flip-flop?
14T (master-slave using TGs): 4 TGs (8T) + 2 feedback inverters (4T) + CLK inverter (2T). Master transparent when CLK=0, slave when CLK=1 → edge-triggered.
Q: What happens to power at smaller technology nodes?
Dynamic drops (lower VDD, CL). But leakage explodes — thinner oxide = gate tunneling, shorter channel = subthreshold leakage. Below ~45nm, leakage ≈ dynamic power. Solutions: high-k dielectrics, FinFET, multi-Vth, power gating.

Charging (step from 0 to V)

V_C(t) = V(1 - e^(-t/τ)) where τ = RC I(t) = (V/R) e^(-t/τ) t = τ → V_C = 0.632V (63.2%) t = 3τ → V_C = 0.950V (95%) t = 5τ → V_C = 0.993V (≈100%)

Derivation (mention in interview to show depth)

KVL: V_in = V_R + V_C = RC·(dV_C/dt) + V_C → First-order linear ODE: dV_C/dt + V_C/RC = V_in/RC → Solution with V_C(0) = 0: V_C(t) = V_in(1 - e^(-t/RC)) Units check: RC = Ω × F = (V/A)(C/V) = C/A = seconds ✓

Case 1: T >> τ (slow switching)

V_C: ________ ________ / \ / \ / \ / \ ____/ \____/ \____ Cap fully charges to V, fully discharges to 0. Almost a square wave with rounded edges.

Case 2: T ≈ τ (comparable)

V_C: /\ /\ /\ / \ / \ / \ / \ / \ / \ ___/ \/ \/ \___ Partial charge/discharge. Clear exponential curves. Oscillates between two intermediate levels after settling.

Case 3: T << τ (fast switching)

V_C: ─────────────────────────── small ripple ≈ V/2 DC level Cap barely responds. Output is nearly the average of input (V/2). This IS low-pass filtering in action.
Interview power move: "The behavior depends on the ratio of RC time constant to the input period. This is fundamentally what makes this circuit a low-pass filter — when the frequency is much higher than 1/(2πRC), the capacitor can't follow, and the output is just the DC average."
R Output across C = LPF Vin ──[===]──┬── Vout │ === C │ GND
H(jω) = 1 / (1 + jωRC) = 1 / (1 + j·f/f_c) |H| = 1 / √(1 + (ωRC)²) ∠H = -arctan(ωRC) f_c = 1/(2πRC) ← -3dB cutoff (|H| = 1/√2 = 0.707)
  • Low freq (f << f_c): Z_C = 1/(jωC) → ∞, so V_out ≈ V_in. Passes through.
  • High freq (f >> f_c): Z_C → 0, C acts as short, V_out → 0. Blocked.
  • Roll-off: -20 dB/decade = -6 dB/octave (first-order)
C Output across R = HPF Vin ──||──┬── Vout │ [R] │ GND
H(jω) = jωRC / (1 + jωRC) |H| = ωRC / √(1 + (ωRC)²) Same f_c = 1/(2πRC)
  • Low freq: C blocks → no current → V_out ≈ 0
  • High freq: C passes → all voltage on R → V_out ≈ V_in

Rule: Look at what element the output is taken across:

  • Output across capacitor → Low-Pass (cap charges slowly, smooths out fast changes)
  • Output across resistor (with cap in series) → High-Pass (resistor sees current only during transitions)
  • Output across inductor → High-Pass (inductor opposes change, passes high freq)
  • Output across resistor (with inductor in series) → Low-Pass

Physical intuition test: At DC (f=0), does the output exist? If yes → low-pass. At very high frequency, does the output exist? If yes → high-pass.

Op-Amp Circuits

From your Analog Computation project + relevant for TI analog division

Two Golden Rules (negative feedback)

  1. No input current: I+ = I- = 0 (infinite input impedance)
  2. Virtual short: V+ = V- (infinite open-loop gain drives the difference to zero)

Inverting Amplifier

V_out = -(R_f / R_in) × V_in Input impedance = R_in (relatively low) Virtual ground at (–) terminal

Non-Inverting Amplifier

V_out = (1 + R_f / R_in) × V_in Input impedance = very high (signal at + terminal) Minimum gain = 1 (unity-gain buffer when R_f = 0)

Summing Amplifier (Inverting)

V_out = -R_f × (V1/R1 + V2/R2 + V3/R3)

Integrator

V_out = -(1/RC) ∫ V_in dt Replaces R_f with capacitor C. This is what you used for Lotka-Volterra!

Differentiator

V_out = -RC × dV_in/dt Replaces R_in with capacitor C.

Lotka-Volterra Equations (Predator-Prey Model)

dx/dt = ax - bxy (prey: grows naturally, eaten by predators) dy/dt = -cy + dxy (predator: dies naturally, grows by eating prey)

Mapping to Op-Amp Circuits

Each equation is implemented as an integrator whose input is formed by summers and multipliers:

For dx/dt = ax - bxy: x(t) ──[×a]──┐ ├──[summer]──[integrator]──→ x(t) (feedback loop) x×y ──[×(-b)]┘ The integrator outputs x(t) = ∫(ax - bxy)dt The multiplier computes x×y using a log/antilog or Gilbert cell circuit The summer adds the two terms

Result: Connect both equation circuits in a loop, set initial conditions using voltage sources, and the voltages oscillate as predator-prey populations. You validated the output waveforms in LTspice against the mathematical solution.

  • Finite open-loop gain (A): Real op-amps have A ≈ 10⁵-10⁶, not ∞. Causes gain error.
  • Finite bandwidth (GBW): Gain × bandwidth = constant. Higher gain → lower bandwidth.
  • Input offset voltage: Small DC voltage between inputs needed to make V_out = 0.
  • Slew rate: Maximum rate of output change (V/μs). If input changes faster, output distorts.
  • CMRR: Common-Mode Rejection Ratio. How well the op-amp ignores signals common to both inputs. Ideal = ∞, real = 80-120 dB.
  • Input bias current: Small current flowing into inputs (nA range). Matters for high-impedance sources.

1. OpCode Decoder (opcode_decoder.vhd)

  • Takes 8-bit opcode I[15:8] from instruction
  • Primary decoder: 4-to-16 using I[15:12] (CASE statement)
  • Nested secondary decoder for INPUT group (I[9:8]) and BRANCH group (I[9:8])
  • Outputs: 23 one-hot lines (exactly one active per instruction) + 4 register select bits (X1, X0, Y1, Y0)
  • 27 outputs total fed into control logic
Why one-hot encoding? • Simple: each instruction just sets 1 bit → no further decode needed in control logic • Control logic becomes OR of instruction lines (sum-of-products) • Trade-off: uses 23 wires vs 5-bit binary (needs 2^5 decode) • For 23 instructions, the area cost is acceptable • Used in your FPGA: FPGA LUTs make one-hot efficient

Q: Why one-hot encoding instead of binary encoding for opcodes?
One-hot: Each instruction sets exactly 1 of 23 lines. Control logic becomes simple OR gates (sum-of-products). No need for priority encoder or complex decode cascading. Trade-off: more wires (23 vs 5 for binary). But for 23 instructions, the area overhead is small, and on FPGAs one-hot is actually preferred because FPGAs have abundant flip-flops but limited routing — one-hot uses more FFs but simpler routing and faster decode.

Binary encoding would use 5 bits (25=32 > 23) but require a complex decoder with AND gates for every instruction. Denser in ASIC, slower to decode.
Q: Why is this a single-cycle design? What are the trade-offs vs. pipelined?
Single-cycle: Each instruction completes in exactly 1 clock cycle. The clock period must accommodate the longest instruction path (e.g., LOAD: PC → Code Memory → Decode → Reg Read → ALU (address calc) → Data Memory → Writeback Mux → Reg Write). This is the critical path.

Trade-off: Simple control (no pipeline hazards, no forwarding, no stalling). But clock speed is limited by worst-case instruction. A pipelined design could run faster by overlapping fetch/decode/execute/writeback of different instructions — but introduces hazards (data, control, structural) requiring forwarding and stalling logic.

For bubble sort on a small CPU, single-cycle is the right choice — complexity of pipelining isn't justified for 23 instructions.
Q: How do the branch conditions work? How does CMP + BRG detect "greater than" for signed numbers?
CMP RX, RY computes RX − RY internally (same as SUB) but discards the result — only updates the 4 flags (C, O, N, Z).

Then: BRG (Branch if Greater) checks: ZF=0 AND (NF XNOR OF)
• ZF=0: result wasn't zero (so not equal)
• NF XNOR OF: Negative flag matches Overflow flag. If no overflow, N=0 means positive result (A>B). If overflow occurred, the sign flipped, so N=1 with O=1 still means A>B.
This is identical to how x86 implements JG (Jump if Greater): ZF=0 AND SF=OF.
Q: Walk me through what happens when ADD RX, RY executes.
Cycle-by-cycle trace for ADD:
1. Fetch: Code Memory reads instruction at PC address → 16-bit instruction on bus
2. Decode: Opcode Decoder sees I[15:12]=0100 → asserts ADD=1 (one-hot). Passes X1,X0 (destination RX) and Y1,Y0 (source RY) from I[11:8]
3. Control: ADD=1 activates: c4,c5=X (read RX on Port0), c6,c7=Y (read RY on Port1), c12=1,c13=0 (ALU=ADD), c14=1 (flags write enable), c10=1 (reg write enable), c8,c9=X (write back to RX)
4. Execute: ALU receives Port0 (RX value) and Port1 (RY value), computes sum, produces result + C,O,N,Z flags
5. Writeback: ALU result passes through muxes (c15=0→ALU, c18=0→ALU not DMEM) → written to RX in register file on rising clock edge. Flags stored in Flags register (c14=1).
6. PC Update: c2=0 (no branch) → PC ← PC+1. Next instruction fetched.
Q: How did you handle the LOADF instruction (load with offset)?
LOADF RX, RY, ADDR: loads RX from DataMem[RY + ADDR].
The control logic sets c11=1 (ALU source mux selects register value, not immediate). ALU performs addition (c12=1, c13=0) of RY's value + the address from instruction bits. The ALU result becomes the data memory read address. The data memory output is routed back through the writeback mux (c18=1) to the register file. This enables array indexing — RY acts as a base/index register, critical for the bubble sort inner loop.
Q: What was the biggest challenge?
Getting the control signal equations right. With 18 control signals and 23 instructions, the control logic truth table is large. A single wrong bit means one instruction corrupts data for a completely different instruction. Debugging required: (1) isolating individual instructions in the testbench, (2) tracing every mux and enable signal in the ModelSim waveform, (3) cross-referencing with the control decode table from the spec. We caught a bug where the SUB overflow formula was using the wrong XOR structure — fixed it to (A[7] XOR B[7]) AND (A[7] XOR result[7]).
Q: If you had more time, what would you improve?
Three things: (1) Pipeline it — add pipeline registers between fetch/decode/execute/writeback stages, implement forwarding for data hazards, and a branch predictor for control hazards. (2) Expand the ISA — add AND/OR/XOR logic ops, multiply, and interrupt support. (3) FPGA hardware verification — we simulated in ModelSim but could map it to an actual FPGA board with switches for input and LEDs/seven-segment displays for output, connecting to physical memory.

Asynchronous (Ripple) Counter

CLK ──→ [T FF₀] ──Q₀──→ [T FF₁] ──Q₁──→ [T FF₂] ──Q₂──→ [T FF₃] ──Q₃ (LSB) (MSB) Each FF clocked by the previous FF's output → delay accumulates! Total delay = N × t_prop → limits max frequency, causes glitches
  • Simple: each T flip-flop (T=1) clocked by output of previous one
  • Problem: propagation delay accumulates — for 4 bits, output valid only after 4 × tprop
  • Glitches: intermediate states appear briefly during ripple — can cause false triggers in downstream logic

Synchronous Counter

CLK ──→ [JK FF₀] ──Q₀ │ ──→ [JK FF₁] ──Q₁ J₁ = Q₀ │ ──→ [JK FF₂] ──Q₂ J₂ = Q₀·Q₁ │ ──→ [JK FF₃] ──Q₃ J₃ = Q₀·Q₁·Q₂ └──── All FFs share the same clock!

Q: What's the difference between synchronous and asynchronous reset?
Synchronous reset: Reset is checked only at the clock edge (if rising_edge(clk) then if rst='1' then Q<='0'). Cleaner timing, no reset glitches, but requires an active clock.
Asynchronous reset: Reset takes effect immediately, regardless of clock (process(clk, rst) if rst='1' then Q<='0' elsif rising_edge(clk)...). Works without clock, but can cause metastability if released near a clock edge. Best practice: assert asynchronously, de-assert synchronously (async assert, sync deassert).

What is Goertzel?

Computes the power at a single specific frequency from N samples, without computing the full FFT. Computationally much cheaper than FFT when you only need a few frequency bins. O(N) per frequency vs O(N log N) for full FFT.

Algorithm for each sample x[n]: s0 = x[n] + coeff * s1 - s2 s2 = s1 s1 = s0 After N samples: Power = s1² + s2² - coeff * s1 * s2 coeff = 2 * cos(2π * k / N) where k = target bin index

You implemented this on the 8051 for tone detection at 60 Hz, 350 Hz, 410 Hz, and 470 Hz with Fs = 2000 Hz and N = 200 samples. Both floating-point and Q8 fixed-point versions.

What is UART?

Universal Asynchronous Receiver-Transmitter — a hardware block that converts parallel data to/from serial. It is asynchronous (no shared clock), so both sides must agree on baud rate beforehand.

  • Start bit: Always LOW (0) — signals beginning of frame
  • Data bits: 5 to 9 bits, LSB first (most common: 8 bits)
  • Parity bit (optional): Even or odd parity for error detection
  • Stop bit(s): Always HIGH (1) — 1 or 2 stop bits
  • IDLE state: Line is HIGH when no data is being transmitted

Throughput Calculation

8N1 at 9600 baud: Each frame = 1 start + 8 data + 1 stop = 10 bits Raw rate = 9600 bits/sec Effective = 9600/10 = 960 bytes/sec Efficiency = 8/10 = 80% 8E1 at 115200 baud: Each frame = 1 start + 8 data + 1 parity + 1 stop = 11 bits Effective = 115200/11 = 10,472 bytes/sec Efficiency = 8/11 = 72.7%

Common Baud Rates

9600, 19200, 38400, 57600, 115200 (most common for debug), 230400, 460800, 921600

Baud rate mismatch: If devices use different baud rates, data is completely garbled. There's no error indication — you just get garbage bytes. This is the #1 debugging issue with UART. Also ensure both sides match data bits, parity, and stop bits.

Flow Control

  • None: Most common in embedded. Risk of buffer overflow if receiver is slower.
  • Hardware (RTS/CTS): Extra wires. RTS = "I'm ready to receive", CTS = "You can send". Prevents overflow.
  • Software (XON/XOFF): Special characters (0x11, 0x13) sent in-band. Can't use if data contains these values (binary data).

Why Open-Drain with Pull-ups?

  • No bus contention: If two devices drive simultaneously, line goes LOW (wired-AND) — no short circuit, no damage
  • Multi-master arbitration: If a master sees SDA LOW when it sent HIGH, it lost arbitration → backs off gracefully
  • Clock stretching: Slow slave holds SCL LOW to pause master — built-in flow control
Pull-up resistor trade-off: Too large (10kΩ+) → slow rise time (RC delay with bus capacitance) → may not reach VIH before next clock edge → FAIL Too small (1kΩ-) → higher power consumption → open-drain sinks more current to pull LOW Rule of thumb: 4.7kΩ for 100kHz, 2.2kΩ for 400kHz, 1kΩ for 1MHz Max bus capacitance: 400pF (limits cable length and device count)

Advanced I2C Features

  • Repeated START (Sr): START without preceding STOP. Used to change direction (write register address → read data) without releasing the bus. Prevents another master from stealing the bus between operations.
  • 10-bit addressing: Two address bytes (first byte: 11110 + A9:A8 + R/W). Allows 1024 addresses. Backwards compatible.
  • General call (0x00): Broadcast to all slaves. Used for software reset, address programming.
  • Clock stretching: Slave holds SCL LOW when it needs more time. Master must check SCL before proceeding. Not all masters support this!

CAN Error Detection (5 mechanisms!)

MechanismHow it works
CRC check15-bit CRC polynomial calculated over message. Receiver computes same CRC and compares.
Bit stuffingAfter 5 consecutive same bits, a complementary bit is inserted. Receiver removes it. Violation = error.
ACK checkTransmitter sends recessive ACK slot. At least one receiver must drive dominant. No ACK = error.
Frame checkFixed-format fields (EOF, delimiters) must have specific values. Violation = error.
Bit monitoringTransmitter reads back the bus. If sent bit ≠ bus bit (outside arbitration) = error.

CAN Error States

Each node maintains two error counters: TEC (Transmit) and REC (Receive) Error Active: TEC < 128 AND REC < 128 → Can transmit error frames with 6 DOMINANT bits (active error flag) → Normal operation Error Passive: TEC ≥ 128 OR REC ≥ 128 → Sends error frames with 6 RECESSIVE bits (passive error flag) → Must wait extra time after transmitting before retrying → Reduced bus impact Bus Off: TEC ≥ 256 → Node disconnects from bus completely → Can only recover after 128 × 11 recessive bits (takes ~1.3ms @1Mbps) → Faulty node isolated automatically!

  • Larger payload: Up to 64 bytes per frame (vs 8 bytes classical)
  • Same arbitration: Uses classical CAN speed for arbitration, then switches to higher speed for data
  • Backwards compatible: CAN FD nodes can coexist with classical CAN nodes (with limitations)

Why differential is better for long distances: Both wires pick up the same noise (common-mode). The receiver takes the DIFFERENCE, canceling the noise. RS-485 Direction Control

RS-485 is half-duplex on 2 wires: nodes share A and B lines. Only one node can transmit at a time! Transceiver has DE (Driver Enable) and R̄E (Receiver Enable) pins: To TRANSMIT: DE=1, R̄E=1 (driver on, receiver off) To RECEIVE: DE=0, R̄E=0 (driver off, receiver on) Firmware must toggle DE/RE before and after transmission. Quick-Fire Interview Q&A RAPID REVIEW

Q: What happens if you forget pull-up resistors on I2C?
The bus lines float at indeterminate voltages since all I2C outputs are open-drain. Communication is completely unreliable — you'll see random ACK/NACK, corrupted data, or no communication at all. The lines might work intermittently due to parasitic capacitance, which makes it extra confusing to debug.
Q: Can you put SPI and I2C devices on the same bus?
No. They are completely different protocols with different electrical characteristics. SPI uses push-pull drivers; I2C uses open-drain. SPI has no addressing; I2C uses addresses. They need separate buses. However, a single microcontroller can have both an SPI peripheral and an I2C peripheral, talking to different devices on different pins.
Q: Why does CAN need 120Ω termination?
The CAN bus is a transmission line. Without termination matching the cable's characteristic impedance (~120Ω), signals reflect off the open ends, causing interference and bit errors. Two 120Ω resistors at each end of the bus (total 60Ω between CANH and CANL) absorb the signal energy. You can verify with a multimeter: measure resistance between CANH and CANL — should read ~60Ω with both terminators present.
Q: SPI has no error detection. How do you ensure data integrity?
Add software-level CRC to your SPI transactions. Some SPI devices (like SD cards) include CRC in their protocol. You can also read-back-verify: write a value, then read it back and compare. For critical systems, use hamming codes or duplicate messages. Alternatively, choose a protocol with built-in error detection (CAN, I2C ACK) if reliability is paramount.
Q: What is the maximum number of I2C devices you can connect?
Theoretically 128 (7-bit addressing: 27 = 128, minus reserved addresses ≈ 112 usable). Practically limited by bus capacitance (max 400pF). Each device and wire adds capacitance. With 20+ devices and long wires, you may exceed 400pF, requiring I2C bus expanders or buffers (like PCA9600). With 10-bit addressing: up to 1024 devices.
Q: How is CAN different from I2C at the physical level?
CAN: Differential signaling (CANH-CANL), works over long distances (40m+), noise-immune, 120Ω termination, dominant/recessive bus states. I2C: Single-ended open-drain with pull-ups, short distances only (<1m), susceptible to noise, wired-AND bus logic. CAN is designed for hostile automotive environments; I2C is for short PCB-level connections.
