Smart Rural Bus Tracking System — End-to-End Information Flow

Bus → Network → Server → Passenger, with SMS as the fallback channel.

Sections: Problem Flow Overview Bus Unit Data Packet Transport Server Passenger Access SMS Fallback Offline Logic Reliability Sources

1. The Problem We're Solving [1] [6]

Rural passengers don't know the real-time location, arrival time, route, or delay status of buses. The core issues:

  • Poor mobile internet at rural stops and along routes.
  • Many passengers use feature phones, not smartphones.
  • Buses go off-grid for stretches — GPS data can't be uploaded live.
  • System must be ultra-low cost (~₹4,000/bus) to scale across ~22,800 government buses.
  • Illiteracy and elderly users — need voice/SMS/display board options.
Core goal: Bus tracking + low-connectivity fallback + ETA prediction, accessible via PWA, SMS, IVR, and bus-stop display boards.

3. The Bus Unit (Edge Device) [2] [3] [5] [7]

A small palm-sized box installed in each bus. Total cost ≈ ₹4,000 per unit; ~₹9–11 Crore for full Tamil Nadu fleet (22,800 buses across 38 districts) [2].

Note on MCU choice: The PDF lists ESP32 in the initial hardware section [1] and MSPM0G3507 in the detailed feasibility and prototype-flow sections [2] [4]. This writeup uses MSPM0G3507 as it is the final selected MCU. Similarly, the comms module is listed as both SIM7600 [2] and A7670 [5] — A7670 is the cheaper LTE option chosen in the cost/size table.

MCU responsibilities

  • Read GPS coordinates over UART, parse NMEA.
  • Track IR-sensor state machine to update occupancy.
  • Read battery voltage via voltage divider + ADC.
  • Build a structured JSON packet.
  • Check network availability; send if up, buffer if down.
  • Resync stored packets when network returns.
  • Detect tamper, GPS failure; emit heartbeat.

Single canonical record. Same shape on every channel (HTTP, SMS-decoded, buffer replay).

  "bus_id":     "BUS_01",
  "route_id":   "ROUTE_05",
  "latitude":   17.9785,
  "longitude":  79.5941,
  "speed":      35,
  "timestamp":  "2026-06-12T10:14:23Z",
  "battery":    "82%",
  "occupancy":  43,
  "health":     "OK"
}

5. Transport — How the Packet Reaches the Server [3] [6]

Primary path: HTTP POST over 4G/GPRS

MCU ──UART──> A7670 ──LTE──> Internet ──> Cloud API
                              POST /api/ingest
                              Content-Type: application/json
                              Body: { bus_id, lat, lng, ... }

Update interval: ~30 seconds (≈ 760 updates/sec across 22,800 buses — light load for cloud).

Fallback path: SMS (when data fails)

MCU ──UART──> A7670 ──SMS via GSM tower──> SMS Gateway ──webhook──> Cloud API

Same logical packet, encoded compactly to fit 160 chars.

Last-resort: local buffer

If both fail, packets are written to flash/SD with a sequence number and replayed when network returns.

6. Cloud Server [2] [3] [6]

Single centralized cloud — not per-district. Cheaper, simpler to scale.

Ingest pipeline

  1. HTTP endpoint POST /api/ingest — accepts JSON from the bus.
  2. SMS webhook POST /api/sms-ingest — gateway forwards decoded SMS.
  3. Both call a single normalizer that produces the canonical record with source: "http" | "sms" | "replay".
  4. Write to time-series DB, keyed by (bus_id, timestamp).
  5. Update in-memory latest position cache for fast PWA reads.

ETA prediction

  • Phase 1: distance to next stop ÷ average route speed.
  • Phase 2: incorporate time-of-day patterns, historical travel times, market-day / school-hour congestion.

Fallback prediction (when live GPS missing)

Algorithm:
1. Server stores fixed route + average travel times between stops.
2. On each update, refresh last_known_position and last_update_time.
3. If (now − last_update_time) > threshold:
   - move bus forward along the route by avg_speed × elapsed
   - mark as "Estimated Location"
4. If silent too long → show "Last Known Location" instead.
5. When network returns → reconcile predicted vs actual,
   correct trajectory, update prediction-accuracy metric.

8. SMS as Fallback Transport (Bus → Server) [6]

This is the heart of the low-connectivity strategy. When the LTE data channel is down but the tower is still reachable, SMS keeps data flowing.

Why SMS works as fallback

  • Rides the GSM control channel — works wherever the SIM can register, even where data fails.
  • No APN, no DNS, no TLS handshake — fewer ways to break.
  • Tiny but sufficient: 160 chars holds a compact bus packet.

Decision logic on the MCU

attempt_send(packet):
    if GPRS_attach() fails after 3 retries        → SMS fallback
    if HTTP POST returns non-2xx (3 retries)      → SMS fallback
    if no ACK within 20 seconds                   → SMS fallback
    after 3 consecutive HTTP failures             → skip HTTP for 1 hour
    else                                          → success on HTTP

Compact payload for SMS

FormatExampleLength
Full JSON{"bus_id":"BUS_01","lat":17.9785,"lng":79.5941,...}~120 chars
Compact pipev1|BUS_01|17.9785|79.5941|35|82|43|1718193263~45 chars

Server-side SMS handling

  1. SMS gateway (MSG91 / Twilio / Exotel) receives the SMS on a long code.
  2. Posts a webhook to /api/sms-ingest with sender + body.
  3. Normalizer parses v1|... back into the canonical JSON.
  4. Record enters the same pipeline as HTTP packets — tagged source: "sms".

9. Offline-First Logic on the Bus [2] [4]

Network Available  → Send packet over HTTP
       ↓
Network Lost       → Try SMS fallback
       ↓
SMS Lost too       → Write to flash / SD with sequence number
       ↓
Network Returns    → Replay buffered packets in order
                     Server dedupes by (bus_id, seq)

Tracking continues uninterrupted; no data is lost even in long dead zones.

10. Reliability Patterns [3] [6]

  • Sequence numbers — monotonic per bus; server detects gaps and dedupes.
  • Idempotency — server keys ingest on (bus_id, seq); replay is safe.
  • Batching in SMS mode — accumulate to 1 SMS every few minutes, not per reading.
  • Daily SMS cap — firmware enforces max SMS/day to bound cost.
  • Heartbeat — one packet every 24 h even when nothing changes, so dead devices are visible.
  • Tamper detection — flagged in health field; raises dashboard alert.
  • Prediction reconciliation — when GPS returns, compare predicted vs actual; feed accuracy metric.

11. Summary

Buses collect GPS + occupancy + health → MCU builds a JSON packet → A7670 sends over 4G (primary) or SMS (fallback) or buffers to flash if both fail → Cloud server normalizes all sources into one pipeline, runs ETA + fallback prediction → Passengers query via PWA, SMS, IVR, or display boards → Authority watches the fleet on a dashboard.

The SMS path is what keeps the system honest in rural India: when data dies, control-channel messaging still gets bus locations to the server, and rural feature-phone users still get ETAs back.
