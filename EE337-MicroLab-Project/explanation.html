I2C Communication Driver

i2c.c / i2c.h — The foundation layer that enables talking to both the BMP280 and DS1307

CONCEPT

How I2C Works — The Protocol

I2C is a 2-wire serial bus where one master (our microcontroller) talks to multiple slaves (BMP280, DS1307) using just two lines:

LineNamePurpose
SDASerial DataCarries data bits (bidirectional)
SCLSerial ClockClock signal (always driven by master)

Every I2C transaction follows this pattern:

SDA: ~~\___[A6][A5][A4][A3][A2][A1][A0][R/W][ACK][D7][D6]...[D0][ACK]___/~~
         |                                   |                        |
SCL: ~~~~|_/~\_/~\_/~\_/~\_/~\_/~\_/~\_/~\_/~|_/~\_/~\_..._/~\_/~\_/~|~~~~
         |                                   |                        |
       START                              data byte                  STOP
START condition: SDA goes LOW while SCL is HIGH.
STOP condition: SDA goes HIGH while SCL is HIGH.
During normal data transfer, SDA only changes when SCL is LOW.

SSDAT Register (I2C Data)

An 8-bit register. Write a byte to it before sending; read from it after receiving.

I2C

Low-Level Functions

i2c.c lines 4–48
void I2C_Init(void)
SSCON = 0xC5;   // binary: 1100 0101
Breaks down as:
  • CR1 = 1, SSEN = 1 → I2C peripheral enabled
  • STA = 0, STO = 0 → no START/STOP yet
  • SI = 0 → interrupt flag cleared
  • AA = 1 → ACK enabled by default
  • CR0 = 0 → along with CR1, sets clock rate
void I2C_Start(void)
Generates a START condition on the I2C bus: 
SSCON = SSCON | 0x20;    // Set STA bit (bit 5) = 1
SSCON = SSCON & 0xF7;    // Clear SI bit (bit 3) = 0  → releases bus
while((SSCON & 0x08)==0); // Wait until SI = 1 (START completed)
SSCON = SSCON & 0xDF;    // Clear STA bit (don't repeat START)

Why clear STA at the end? If you leave STA = 1, the hardware would try to send another START after the next operation. You only want one START per transaction.

void I2C_Stop(void)
Generates a STOP condition: 
SSCON = SSCON | 0x10;    // Set STO bit (bit 4) = 1
SSCON = SSCON & 0xF7;    // Clear SI = 0  → triggers STOP
while(SSCON & 0x10);     // Wait until STO auto-clears
Unlike STA, the STO bit clears itself automatically once the STOP condition is complete. That's why we wait for it to become 0.
void I2C_Write(unsigned char byte)
Sends one byte (address or data) on the bus: 
SSDAT = byte;            // Load byte into data register
SSCON = SSCON & 0xF7;    // Clear SI → hardware starts sending
while((SSCON & 0x08)==0); // Wait until SI = 1 (byte sent + ACK/NACK received)

After this, the slave has either ACKed or NACKed. The status code in SSCON indicates which, but this code doesn't check it (simplified).

unsigned char I2C_Read(void)
Reads one byte from the bus: 
SSCON = SSCON & 0xF7;    // Clear SI → hardware clocks in 8 bits
while((SSCON & 0x08)==0); // Wait until SI = 1 (byte received)
return SSDAT;            // Return the received byte
Important: You must call I2C_Ack() or I2C_Nack() before calling I2C_Read(). The AA bit tells the hardware what to send on the 9th clock pulse (the acknowledge slot).
void I2C_Ack(void) / void I2C_Nack(void)
// ACK: "I want more data"
SSCON = SSCON | 0x04;    // AA = 1 → will pull SDA LOW after read

// NACK: "This is the last byte"
SSCON = SSCON & 0xFB;    // AA = 0 → will leave SDA HIGH after read
void I2C_RepeatedStart(void)

Identical to I2C_Start(). A Repeated START is a START sent without a preceding STOP. It's used when the master needs to change direction (write → read) without releasing the bus. This prevents another master from stealing the bus between transactions.

High-Level Device Functions

i2c.c lines 51–88

These combine the low-level primitives into complete I2C read/write transactions.

unsigned char I2C_ReadRegister(devAddr, regAddr)

Reads one byte from a specific register of a slave device. This is a two-phase operation:

Phase A — Tell the slave WHICH register:

START
↓
Write: devAddr << 1 | 0 = address + WRITE bit
↓
Write: regAddr = register pointer

Phase B — Read the data:

Repeated START (switch from write → read)
↓
Write: devAddr << 1 | 1 = address + READ bit
↓
NACK (only 1 byte needed) → Read byte from SSDAT
↓
STOP
Why two phases? I2C devices have many registers. You first WRITE the register address (telling the slave "point your internal pointer to this register"), then READ the data from that register. The Repeated START separates these two directions.
void I2C_WriteRegister(devAddr, regAddr, val)

Writes one byte to a specific register. Simpler because both the register address and data go in the same direction (master → slave):

START
↓
Write: devAddr << 1 | 0 (address + WRITE)
↓
Write: regAddr (register address)
↓
Write: val (data to store)
↓
STOP
void I2C_ReadMultiple(devAddr, regAddr, *buf, len)

Reads len consecutive bytes starting from regAddr into buffer buf[].

  • Same two-phase approach as I2C_ReadRegister
  • But reads multiple bytes in a loop
  • Sends ACK after every byte except the last
  • Sends NACK before the last byte → tells slave "stop after this"
  • The slave auto-increments its internal register pointer, so consecutive reads give consecutive registers
for(i = 0; i < len; i++){
    if(i < len-1)  I2C_Ack();   // More bytes to come
    else           I2C_Nack();  // Last byte
    buf[i] = I2C_Read();
}

lcd.h — Controls a 16x2 HD44780 character LCD in 8-bit parallel mode

CONCEPT

How the LCD Works

The HD44780 LCD has an internal controller with its own memory. You send it commands (to control cursor, clear screen, etc.) and data (ASCII characters to display). It distinguishes between them using the RS pin.

Pin Connections

RWP0.1Read/Write: always 0 (we only write)

D0–D7P2.0–P2.78-bit parallel data bus

How a Write Cycle Works

RS:  ____/~~~~~~~~~~~~~~~~~\____    (0 for cmd, 1 for data)
RW:  ___________________________    (always 0 = write)
EN:  ________/~~~~~~~~~~\______     (pulse HIGH then LOW)
P2:  ========[  DATA BYTE  ]====    (stable while EN is HIGH)
              ^             ^
              |             |
         EN goes HIGH   EN goes LOW
         (LCD reads)    (data latched)

void msdelay(unsigned int time)

Software delay function. The inner loop (j < 382) is calibrated to produce approximately 1 ms delay per iteration of the outer loop at the crystal frequency used.

for(i=0; i<time; i++)       // outer loop = 'time' ms
    for(j=0; j<382; j++);   // inner loop ~ 1 ms
void lcd_cmd(unsigned int i)

Sends a command byte to the LCD:

RS = 0;       // Command mode (not data)
RW = 0;       // Write mode
EN = 1;       // Enable HIGH
P2 = i;       // Put command byte on data bus
msdelay(10);  // Hold for LCD to read
EN = 0;       // Enable LOW → LCD latches the command
void lcd_write_char(unsigned char ch)

Sends a data byte (ASCII character) to display:

RS = 1;       // Data mode (character, not command!)
RW = 0;       // Write mode
EN = 1;       // Enable HIGH
P2 = ch;      // Put ASCII character on data bus
msdelay(10);  // Hold
EN = 0;       // Latch → character appears on screen

The only difference from lcd_cmd() is RS = 1 instead of RS = 0.

void lcd_write_string(unsigned char *s)

Takes a pointer to a null-terminated string and prints each character:

while(*s != '\0')        // until null terminator
{
    lcd_write_char(*s++);  // print char, advance pointer
}

The cursor auto-increments (thanks to 0x06 entry mode), so characters appear left-to-right.

void lcd_print_num(int num)

Prints an integer number on the LCD. Since there's no printf() on embedded systems, we manually extract digits:

1Handle num == 0: just print '0'
2Handle negative: print '-', then work with -num
3Extract digits using num % 10 (gives last digit), convert to ASCII by adding '0' (0x30). Store in array.
4Digits come out in reverse order (units first), so print the array backwards

Example: num = 253

UART Serial Communication

serial.h — Interrupt-driven UART for sending data to a PC terminal

CONCEPT

How UART Works

UART is an asynchronous protocol — there's no clock line. Both sides must agree on the baud rate (bits per second) beforehand.

UART Frame Format (Mode 1)

Idle ___      ___________________________________      _____ Idle
        \    |   |   |   |   |   |   |   |   |   |   |
         |ST | D0| D1| D2| D3| D4| D5| D6| D7|STOP|
         |___|___|___|___|___|___|___|___|___|___|

         START  <---- 8 data bits (LSB first) --->  STOP
          bit                                        bit
PartValuePurpose
Start bit0 (LOW)Signals the beginning of a frame
Data bits8 bitsThe actual character (LSB first)
Stop bit1 (HIGH)Signals end of frame, line goes idle

Baud Rate Generation

In the 8051, Timer 1 in Mode 2 (8-bit auto-reload) generates the baud rate clock:

Timer 1 Mode 2

The timer counts up from TH1 to 0xFF, overflows, and automatically reloads TH1. This creates a periodic overflow that clocks the UART.

Baud Rate Formula

Baud = (2^SMOD / 32) × (f_osc / (12 × (256 - TH1)))

TH1 = 0xCB = 203, so 256 - 203 = 53. The exact baud depends on the crystal frequency.

void transmit_char(unsigned char ch)
SBUF = ch;             // Load character into Serial Buffer
                       // → hardware automatically starts transmitting
while(!tx_complete);   // Wait for ISR to set tx_complete flag
tx_complete = 0;       // Clear flag for next transmission

Flow: Writing to SBUF triggers hardware to shift out the byte bit-by-bit. When all bits (start + 8 data + stop) are sent, hardware sets TI = 1, which triggers the ISR. The ISR sets tx_complete = 1. Main code detects this and proceeds.

void transmit_string(unsigned char *s)

Sends a null-terminated string character by character:

while(*s != 0)
    transmit_char(*s++);
unsigned char receive_char(void)
while(!rx_complete);   // Wait for ISR to set rx_complete
rx_complete = 0;       // Clear flag
ch = SBUF;             // Read received byte from Serial Buffer
return ch;

(Not used in this project's main loop, but available for bidirectional communication.)

void serial_ISR(void) interrupt 4

The Interrupt Service Routine for the serial port. Triggered whenever TI or RI is set by hardware.

void serial_ISR(void) interrupt 4
{
    if(TI == 1) {          // Transmit complete?
        TI = 0;            // Clear TI (MUST be done manually!)
        tx_complete = 1;   // Signal to transmit_char()
    }
    else if(RI == 1) {     // Receive complete?
        RI = 0;            // Clear RI (MUST be done manually!)
        rx_complete = 1;   // Signal to receive_char()
    }
}
interrupt 4 = Serial port interrupt vector at address 0x0023.
Critical: TI and RI flags are NOT auto-cleared by hardware. If you don't clear them in the ISR, the interrupt fires endlessly!

DS1307 Real-Time Clock Driver

r2c.c / r2c.h — Keeps accurate time using an external RTC chip on the I2C bus

CONCEPT

About the DS1307

The DS1307 is a battery-backed Real-Time Clock (RTC). It has its own 32.768 kHz crystal oscillator and keeps counting time even when the main system is powered off (via a coin cell battery).

PropertyValue
I2C Address0x68 (7-bit, fixed — cannot be changed)
Data FormatBCD (Binary-Coded Decimal)
Time KeepingSeconds, Minutes, Hours, Day, Date, Month, Year
BackupBattery-backed (CR2032 coin cell)

DS1307 Register Map (Time Registers)

AddressRegisterBit 7Bits [6:4]Bits [3:0]
0x00SecondsCH10s digit (0–5)1s digit (0–9)
0x01Minutes010s digit (0–5)1s digit (0–9)
0x02Hours010s digit (0–2)1s digit (0–9)
CH (Clock Halt) Bit: Bit 7 of register 0x00. When CH = 1, the oscillator is stopped and the clock doesn't tick. You MUST clear CH to 0 to start the clock. That's why R2C_SetTime() masks the seconds with & 0x7F.

BCD Encoding — Deep Dive

BCD stores each decimal digit separately in 4 bits (a nibble). This is different from normal binary!

DecimalBCD (Hex)BCD (Binary)Normal BinarySame?
90x090000 10010000 1001Yes (single digit)
250x250010 01010001 1001NO!
590x590101 10010011 1011NO!
Why BCD? It's easy to display: each nibble is already a decimal digit. No division needed. Older hardware (like 7-segment displays) can decode BCD directly.

RTC Functions Explained

r2c.c
void R2C_Init(void)

Just a 200ms startup delay. The DS1307 needs time to stabilize after power-on before I2C communication begins.

unsigned char BCD_to_Dec(unsigned char bcd)

Converts BCD to normal decimal:

return ((bcd >> 4) * 10) + (bcd & 0x0F);
//      ^^^^^^^^^^^         ^^^^^^^^^^^^
//     upper nibble          lower nibble
//     (tens digit)         (ones digit)

Example: BCD_to_Dec(0x25)

  • 0x25 >> 4 = 0x02 = 2 (tens digit)
  • 0x25 & 0x0F = 0x05 = 5 (ones digit)
  • Result: 2 * 10 + 5 = 25
unsigned char Dec_to_BCD(unsigned char dec)

Converts decimal to BCD:

return ((dec / 10) << 4) | (dec % 10);
//      ^^^^^^^^^^^^        ^^^^^^^^^
//      tens → upper nibble  ones → lower nibble

Example: Dec_to_BCD(25)

  • 25 / 10 = 2 → 2 << 4 = 0x20
  • 25 % 10 = 5
  • Result: 0x20 | 0x05 = 0x25
void R2C_SetTime(sec, min, hour)

Writes the given time to the DS1307:

// Write seconds — clear CH bit (bit 7) to START the clock
I2C_WriteRegister(DS1307_ADDR, 0x00, Dec_to_BCD(sec) & 0x7F);

// Write minutes
I2C_WriteRegister(DS1307_ADDR, 0x01, Dec_to_BCD(min));

// Write hours (24-hour format)
I2C_WriteRegister(DS1307_ADDR, 0x02, Dec_to_BCD(hour));
Why & 0x7F?
0x7F = 0111 1111. ANDing with this clears bit 7 (CH bit) to 0, which means "oscillator ON, clock is running". Without this, the clock might be halted.
void R2C_GetTime(*sec, *min, *hour)

Reads current time from the DS1307 and converts BCD → decimal:

*sec  = BCD_to_Dec(I2C_ReadRegister(DS1307_ADDR, 0x00));
*min  = BCD_to_Dec(I2C_ReadRegister(DS1307_ADDR, 0x01));
*hour = BCD_to_Dec(I2C_ReadRegister(DS1307_ADDR, 0x02));

Uses pointers (*sec, *min, *hour) so the function can return three values at once. The caller passes addresses: R2C_GetTime(&sec, &min, &hour).

BMP280

Calibration & Compensation Formula

Every BMP280 chip is slightly different due to manufacturing variations. To get accurate temperature, you must use factory calibration data stored in the chip's non-volatile memory.

Step 1: Read Calibration Data (done once at startup)

I2C_ReadMultiple(BMP280_ADDR, 0x88, calib, 6);

// Assemble into 16-bit values (LITTLE-ENDIAN: low byte first!)
dig_T1 = (calib[1] << 8) | calib[0];  // unsigned 16-bit
dig_T2 = (calib[3] << 8) | calib[2];  // signed 16-bit
dig_T3 = (calib[5] << 8) | calib[4];  // signed 16-bit
ParameterRegistersTypeRole in Formula
dig_T10x88, 0x89unsigned intOffset correction
dig_T20x8A, 0x8Bsigned intLinear sensitivity
dig_T30x8C, 0x8Dsigned intQuadratic (nonlinearity) correction
Little-Endian: The low byte comes first in memory. Register 0x88 = low byte of dig_T1, register 0x89 = high byte. So we assemble as (high << 8) | low.

Step 2: Read Raw Temperature (every second)

I2C_ReadMultiple(BMP280_ADDR, 0xFA, temp_raw, 3);

// Assemble 20-bit ADC value from 3 bytes:
adc_T = ((long)temp_raw[0] << 12) |   // MSB: bits [19:12]
        ((long)temp_raw[1] << 4)  |   // LSB: bits [11:4]
        (temp_raw[2] >> 4);            // XLSB: bits [3:0]

Why 20 bits from 3 bytes?

  temp_raw[0]     temp_raw[1]     temp_raw[2]
 [  MSB byte  ]  [  LSB byte  ]  [XLSB byte  ]
  b19 .... b12    b11 .... b4     b3 b2 b1 b0 x x x x
                                  ^^^^^^^^^^^^
                                  upper nibble only!

Step 3: Apply Compensation Formula

This formula is straight from the BMP280 datasheet (Section 8.1). It corrects for manufacturing variations using the calibration data:

// var1 = linear correction term
var1 = ((adc_T / 8) - ((long)dig_T1 * 2)) * ((long)dig_T2) / 2048;

// var2 = quadratic (non-linear) correction term
var2 = (adc_T / 16) - (long)dig_T1;
var2 = (var2 * var2) / 4096 * ((long)dig_T3) / 16384;

// Combine and scale
T = (var1 + var2) * 5 + 128;
T = T / 256;

MAIN

Main Loop (runs forever)

main.c lines 87–143
Loop Structure — 1 Hz Update
while (1) {
    R2C_GetTime(&sec, &min, &hour);     // Read current time from RTC

    if(sec != prev_sec) {               // Only update when second changes
        prev_sec = sec;
        // ... do all the work ...
    }
}

Why sec != prev_sec? The while(1) loop runs thousands of times per second. Without this check, we'd:

  • Read the BMP280 thousands of times per second (wasteful, sensor needs time)
  • Clear and redraw the LCD thousands of times (causes flicker)
  • Flood the UART with thousands of messages per second

Instead, we poll the RTC and only act when the second counter changes — giving exactly 1 update per second.

Inside the Loop — Step-by-Step

// Same algorithm: extract digits, store reversed, print forward
void uart_print_num(int num)
{
    unsigned char digits[10];
    int i = 0, j;
    if(num == 0){ transmit_char('0'); return; }
    if(num < 0){ transmit_char('-'); num = -num; }
    while(num > 0){
        digits[i++] = (num % 10) + '0';
        num /= 10;
    }
    for(j = i-1; j >= 0; j--)
        transmit_char(digits[j]);
}

Key Concepts for Your Quiz

  1. I2C Address Byte: 7-bit address is shifted left by 1 bit. Bit 0 = R/W (0=write, 1=read). BMP280 at 0x76: write byte = 0xEC, read byte = 0xED. DS1307 at 0x68: write = 0xD0, read = 0xD1.
  2. I2C Repeated START: Used to switch direction (write → read) without releasing the bus. First you WRITE the register address, then Repeated START, then READ the data. No STOP in between.
  3. ACK/NACK in multi-byte reads: Master sends ACK after every byte except the last one. NACK on the final byte tells the slave to release the data line.
  4. SSCON bits: STA (bit 5) = start, STO (bit 4) = stop, SI (bit 3) = interrupt flag (wait on this!), AA (bit 2) = ack/nack control. SI must be cleared to let bus proceed. STO auto-clears; STA does not.
  5. BCD Encoding: Each decimal digit uses 4 bits. 0x25 = 25 decimal (NOT 37). Convert: (upper nibble × 10) + lower nibble. Used by DS1307 for time storage.
  6. Clock Halt (CH) Bit: Bit 7 of DS1307 seconds register. CH=1 stops the clock. Must clear to 0 (& 0x7F) for the clock to tick. This is done in R2C_SetTime().
  7. BMP280 Forced Mode (0x21): Takes one measurement then returns to sleep. Must write to ctrl_meas EACH time you want a new reading. Contrast with Normal mode (continuous).
  8. BMP280 Calibration: Factory-stored trim values (dig_T1/T2/T3) are unique per chip. Read once at startup from registers 0x88–0x8D. Stored little-endian. Raw ADC is useless without these.
  9. Temperature Compensation: var1 = linear term, var2 = quadratic term. Final T is in 0.01°C units (T=2534 means 25.34°C). Uses integer math only (no FPU on 8051).
  10. 20-bit ADC Assembly: adc_T = (MSB << 12) | (LSB << 4) | (XLSB >> 4). Three bytes combine to form a 20-bit value. The bottom 4 bits of XLSB are unused.
  11. UART Baud Rate: Timer 1 in Mode 2 (8-bit auto-reload). TH1 reload value determines baud rate. Timer overflow rate = baud clock. SCON=0x50 = Mode 1, 8-bit, receiver enabled.
  12. UART ISR (interrupt 4): Both TI and RI trigger the SAME interrupt. ISR checks which flag is set. Flags must be cleared manually in software (not auto-cleared). tx_complete/rx_complete are software flags polled by main code.
  13. LCD RS Pin: RS=0 for commands (clear, cursor move, init). RS=1 for data (characters). EN pin must be pulsed HIGH→LOW to latch data. 0x80 = line 1, 0xC0 = line 2.
  14. LCD Init Sequence: 0x38 (8-bit, 2-line) → 0x06 (auto-increment) → 0x0C (display on, cursor off) → 0x01 (clear) → 0x80 (cursor home). Order and delays matter.
  15. Polling vs Interrupt: The RTC time is polled (main loop continuously reads it). The UART uses interrupts (ISR sets flags, main code waits on flags). Polling is simpler; interrupts are more efficient.
  16. Little-Endian: BMP280 stores calibration data with the low byte at the lower address. So register 0x88 = low byte of dig_T1, 0x89 = high byte. Assemble as (high << 8) | low.
